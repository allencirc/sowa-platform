# ADR 0001: Internationalisation approach

- **Status:** Accepted
- **Date:** 2026-04-05
- **Deciders:** SOWA platform team

## Context

SOWA must serve Ireland's offshore-wind workforce, which includes a
significant number of non-native-English speakers. The tender requires the
platform to support English (default), Irish (`ga`), Polish (`pl`),
Ukrainian (`uk`) and Portuguese (`pt`) at launch, with room to grow.

The app is on **Next.js 16 (App Router) + React 19**. Next 16 renamed
`middleware.ts` to `proxy.ts` and made the `params` object async. All our
pages use Server Components by default.

## Options considered

1. **`next-intl`** — library with ICU message support, typed hooks,
   client+server helpers, built-in locale middleware.
2. **Next.js built-in i18n** — no runtime dependency; use a `[locale]`
   dynamic segment plus `proxy.ts` for detection and a hand-rolled
   `getDictionary()` helper loading JSON files. This is the pattern shown
   in `node_modules/next/dist/docs/01-app/02-guides/internationalization.md`.
3. **`next-international`** / **`paraglide`** — alternative libraries.

## Decision

**Use Next.js built-in i18n with a `[locale]` segment + JSON dictionaries.**

Reasons:

- **Zero new runtime dependencies.** Every dependency we add is one more
  thing to audit, upgrade, and ship past compliance review. For a tender
  build we keep the supply chain small.
- **Clean fit with Next 16 App Router.** The built-in pattern is what the
  Next 16 docs document first-class; `next-intl` v3 works but adds another
  layer on top of `params` (which is already async in 16) and its own
  middleware that we'd have to reconcile with our existing `proxy.ts` (which
  handles READ_ONLY kill-switch and admin auth).
- **Server-first.** Dictionaries are plain JSON imported from server
  components — they never ship to the client, so bundle size is unaffected
  by how many strings we add.
- **Good enough ergonomics.** We don't need ICU plurals or rich formatting
  for the seeded copy (nav, hero, footer, CTAs, question stems). If that
  changes — e.g. once we localise diagnostic gap narratives with
  pluralisation — we can adopt `@formatjs/intl-messageformat` piecemeal
  without restructuring routes.

## Consequences

- URLs are `/[locale]/...`. `proxy.ts` redirects `/` and any un-prefixed
  path to the user's preferred locale (from `Accept-Language`, clamped to
  the supported set, defaulting to `en`).
- The root `<html lang>` is derived from the locale segment. The proxy
  sets an `x-sowa-locale` request header so the root layout (which sits
  above `[locale]`) can read it via `next/headers` without re-parsing the
  URL.
- Admin routes (`/admin/*`) and API routes (`/api/*`) are **not**
  localised. They stay at their current paths. The proxy's locale
  redirect only runs on the public surface.
- Dictionaries live in `messages/{locale}.json`. The English file is the
  source of truth; every key must exist in English. Non-English files may
  carry `[TODO]`-prefixed values as placeholders until translation lands —
  these are greppable and fail-loud in review.
- Links inside the public site must be locale-aware. We export a small
  `localeHref(locale, path)` helper and use it in `Header`, `Footer` and
  anywhere that builds nav. Hard-coded `/careers` links still work because
  the proxy will redirect them, but we prefer the explicit form.

## Revisit triggers

- If we need ICU plurals, gendered strings, or number/date formatting in
  more than a handful of places, adopt `next-intl` and migrate the JSON
  files (they are already ICU-compatible shape).
- If translator tooling (Crowdin, Lokalise) requires a different file
  format, we swap the loader in `src/lib/i18n.ts`; call sites stay the same.
